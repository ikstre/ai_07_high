### 3-3. GPU 런타임 캐시 + on-demand worker unload (신규)

**배경**: 현재 ComfyUI가 systemd로 항상 active라 FLUX weight가 L4 24GB VRAM 중 약 14GB를 상시 점유한다. HyperCLOVA X SEED(HF local) 같은 텍스트 모델을 동시에 올리면 OOM 가능성이 높다. 단순 API 응답 캐시가 아니라, "텍스트 요청이면 텍스트 worker만, 이미지 요청이면 이미지 worker만" 올리고 idle 시 내리는 GPU resident model 관리가 필요하다.

**목표**:
- cache hit이면 어떤 GPU worker도 새로 띄우지 않고 저장된 결과 반환
- text cache miss이면 text worker(HyperCLOVA SEED local server)만 로드
- image cache miss이면 image worker(ComfyUI)만 로드
- GPU exclusive mode에서는 text/image 전환 시 반대 worker를 먼저 stop
- idle timeout(예: 5~10분) 후 worker process를 stop해서 VRAM 반환

**권장 설계**:
1. 신규 모듈 후보: `backend/runtime_workers.py`, `backend/result_cache.py`
2. 런타임 상태/캐시 경로:
   - `data/runtime/worker_state.json`
   - `data/runtime/cache/text/<sha256>.json`
   - `data/runtime/cache/image/<sha256>.json`
3. 환경 변수 후보:
   - `GPU_WORKER_MODE=always_on|on_demand|exclusive` (기본 `always_on` 또는 기존 동작 보존)
   - `GPU_WORKER_IDLE_TIMEOUT_SECONDS=600`
   - `GPU_WORKER_LOCK_PATH=data/runtime/gpu_worker.lock`
   - `TEXT_WORKER_CMD=conda run -n sprint_high python tools/hyperclova_seed_openai_server.py`
   - `TEXT_WORKER_HEALTH_URL=http://127.0.0.1:11501/health`
   - `IMAGE_WORKER_SERVICE=comfyui`
   - `IMAGE_WORKER_HEALTH_URL=http://127.0.0.1:8188/system_stats`
4. 캐시 키:
   - text: normalized ad payload + provider id + model id + prompt/policy version
   - image: `image_prompt` + workflow file content hash + dimensions + model config + seed policy
   - 주의: 현재 ComfyUI workflow는 `{seed}`를 시간 기반으로 치환한다. 캐시 재사용을 하려면 "cache hit은 기존 이미지 반환", "강제 재생성은 seed 새로 생성" 옵션을 분리해야 한다.
5. 요청 처리 순서:
   - cache lookup
   - hit: cached result return, worker start 금지
   - miss: `gpu_worker.lock` 획득
   - `exclusive` 모드면 반대 worker stop
   - 필요한 worker health check → 없으면 start
   - 요청 실행 → 결과 저장 → last_used 갱신
   - lock 해제
   - idle reaper는 request 후 background task 또는 주기적 lazy check로 실행

**구현 위치 후보**:
- text path: `backend/ai.py:generate_copy_experiment()`, `generate_ad_copy()`
- image path: `backend/ai.py:create_image_job()`, `poll_image_job()`, `image_reference_from_job()`
- worker health/start/stop: `backend/runtime_workers.py`
- `start.sh`: ComfyUI를 항상 요구하지 말고 `GPU_WORKER_MODE=on_demand|exclusive`면 "필요 시 start" 안내만 출력

**검증 계획**:
1. 사전 상태:
   ```bash
   nvidia-smi
   systemctl is-active comfyui
   curl -s http://127.0.0.1:8188/system_stats
   curl -s http://127.0.0.1:11501/health
   ```
2. text cache hit 검증:
   - 동일 payload로 `/ai/copy/experiment` 2회 호출
   - 1회차: cache miss, text worker start 가능
   - 2회차: cache hit, worker start/stop 로그 없어야 함
   - 응답에 `cache_hit: true`, `cache_key` 또는 로그 표시 권장
3. image cache hit 검증:
   - 동일 `image_prompt`/workflow/dimensions로 이미지 작업 2회 요청
   - 2회차는 ComfyUI `/prompt` 호출 없이 기존 image job 또는 cached image URL 반환
   - cached image 파일 존재와 poster 합성 정상 확인
4. exclusive worker 전환 검증:
   - ComfyUI active 상태에서 text miss 요청 → ComfyUI stop 후 text worker start 확인
   - text worker active 상태에서 image miss 요청 → text worker stop 후 ComfyUI start 확인
   - 전환 전후 `nvidia-smi` VRAM 점유가 내려갔다가 올라가는지 확인
5. idle unload 검증:
   - `GPU_WORKER_IDLE_TIMEOUT_SECONDS=30` 으로 낮춰 실행
   - 마지막 요청 후 30~60초 내 worker stop 확인
   - `nvidia-smi`에서 해당 python process 및 VRAM 점유 해제 확인
6. 회귀 검증:
   ```bash
   python -m py_compile backend/ai.py backend/runtime_workers.py backend/result_cache.py
   python tools/scan_secrets.py --all
   bash -n start.sh
   curl -s http://127.0.0.1:8010/health
   ```
